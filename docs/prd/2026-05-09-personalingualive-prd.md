# PersonaLinguaLive(PLL)产品需求文档

> 版本:v0.1 · 创建日期:2026-05-09 · 状态:草案
> 本文档为持续迭代文档,所有变更记录于第 9 节"版本变更日志"

---

## 1. 背景与愿景

### 1.1 背景
传统英语学习软件普遍存在两类问题:
- **题海/课程化**:割裂真实语境,学完即忘,缺乏沉浸感
- **AI 对话型(如各类 AI 陪练)**:对话场景虚拟、与现实脱钩,用户难以持续

PersonaLinguaLive(下称 PLL)切入一个差异化点:**让用户身边真实环境中的物体直接成为英语对话伙伴**。一只马克杯、一个键盘、一袋面包都能拟人化开口和你聊它的世界,把学习者的真实生活变成无限延展的语言学习场景。

### 1.2 产品愿景
**Anything you see can teach you English.**
让"看到什么 → 用英语聊什么"成为最自然的学习入口,把英语学习从"打开一个 App 学半小时"变成"指着任意东西就能聊一会儿"。

### 1.3 产品代号语义
- Persona:每个物体拥有独立人设
- Lingua:语言学习
- Live:实时语音对话、所见即可对话

## 2. 用户画像与场景

### 2.1 目标用户
**全年龄英语学习者**(自适应水平),典型用户群:
- **儿童学习者(陪伴模式)**:家长一同使用,把家中物品拟人化为孩子的英语朋友
- **学生**(中学/大学):为兴趣或考试,在书桌、宿舍场景碎片化练习
- **职场学习者**:在咖啡馆、办公室,用真实环境物体进行口语场景训练
- **旅行/留学准备者**:针对出行场景(机场、餐厅)主动构建对话练习

### 2.2 核心使用场景
1. **场景 A — 即兴口语练习(主场景)**:用户拍下眼前桌面/厨房/咖啡馆,点击其中任意物品开口聊天
2. **场景 B — 主题学习**:用户进入内置场景图库(V2),选"机场""动物园"等主题练习对应词汇
3. **场景 C — 通勤碎片**:无相机可用时,从历史图片中复用,继续上次的对话

### 2.3 用户旅程(MVP)
```
打开网页 → 选择/拖拽/拍照上传图片
       → 自动内容安全检查(<3s)
       → 显示带可点击热区的图片
       → 点击某物体 → 物体长出嘴巴,主动开口打招呼
       → 用户按住麦克风说话(或松开持续模式)
       → 文字气泡同屏 + AI 语音回复 + 嘴型同步
       → AI 自然纠错 + 语法解释(可折叠)+ 追问
       → 用户可点其他物体切换角色,或结束对话
       → 对话结束生成本次"语法点 + 生词"摘要,存入本地
```

## 3. 功能范围与版本路线

### 3.1 V1 MVP — 核心主闭环
目标:验证"物体拟人化对话"这个核心体验是否成立,跑通端到端最短路径。

| 编号 | 模块 | 描述 |
|------|------|------|
| M1 | 图片上传 | 拖拽/文件选择/调用摄像头拍照 |
| M2 | 内容安全过滤 | 拒绝含人脸/NSFW/敏感内容图片,只放行物品与场景 |
| M3 | 物体检测与热区 | AI 识别可对话物体并渲染为可点击区域 |
| M4 | 2D 拟人化渲染 | 嘴型贴图 + 基于音频幅度的 lip-sync |
| M5 | 角色人设自动生成 | 每个物体由 LLM 基于"物体+场景"生成独立人设 |
| M6 | 语音对话引擎 | STT + LLM 流式 + TTS,文字气泡同屏 |
| M7 | 深度教学反馈 | 隐性纠错 + 显性错误标注(可折叠)+ 追问练习 |
| M8 | 用户水平自选 | 入门/中阶/高阶三档,影响 AI 用词与纠错颗粒度 |
| M9 | 本地数据持久化 | IndexedDB 存对话历史、生词、偏好;含"清除所有数据"入口 |
| M10 | 对话总结卡 | 单次对话结束后生成"语法点 + 生词"摘要 |

### 3.2 V2 — 学习沉淀 + 内容扩展
| 模块 | 描述 |
|------|------|
| 内置场景图库 | 厨房/动物园/机场/咖啡馆/书桌等预置高质量场景图,无需拍照即可上手 |
| 个人词汇本 | 自动从对话中收集生词,带例句、发音、Anki 式间隔复习 |
| 表情/眼神动态 | 角色不仅嘴巴动,眼睛会眨、注视方向跟随用户(根据光标/语音活跃) |

### 3.3 V3 — 玩法深化 + 社交
| 模块 | 描述 |
|------|------|
| 多角色同场互动 | 同一图中多个物体可互聊,用户可插话(如"蛋糕和锅"对话,用户介入) |
| 任务/闯关系统 | 主题任务(如"在厨房学会 5 个烘焙词汇")、徽章、连续学习日 |
| 发音评分 | 对用户语音做发音评估(音素级或单词级反馈) |
| 口语周/月报 | 数据可视化:对话时长、覆盖话题、生词增长、薄弱发音 |
| 对话片段分享 | 生成可分享的对话卡片(图片+文字+音频片段) |

### 3.4 未来潜力点(暂不展开)
- 家长/教师仪表盘(只读进度)
- 多语言支持(西班牙语、日语等)
- 移动端原生 App / 微信小程序
- AR 实时识别(摄像头实时画面替代静态图)
- 角色记忆持久化(同一物体跨次对话保留人设记忆)

## 4. 详细功能需求(V1)

### 4.1 M1 — 图片上传

**输入方式**
- 拖拽到页面任意位置
- 点击上传区域选择文件
- "拍照"按钮调用 `navigator.mediaDevices.getUserMedia` 取浏览器摄像头

**约束**
- 格式:JPG / PNG / WebP / HEIC(后端转码)
- 大小:原图 ≤ 10MB;客户端先做 max 1600px 压缩再上传
- 单次仅支持一张图

**异常**
- 不支持的格式:友好提示并示例支持的格式
- 超大文件:提示并自动压缩重试

### 4.2 M2 — 内容安全过滤

**审核内容**
- 人脸(任何可识别人脸,包括儿童照片、家庭合照)
- NSFW(色情、暴露)
- 暴力、血腥、武器
- 政治敏感符号
- 文本主体图(防止用户当 OCR 工具)

**实现思路**
- 接入视觉模型一次推理同时返回:`{is_safe, reject_reasons, objects[]}`
- 不安全:整图拒绝,明确告知拒绝类别(如"检测到人脸,本应用仅支持物品和场景类图片")
- 边界场景(如玩具人偶)放行,产品说明里明示规则

**目标指标**
- 拒绝召回率(应拒绝的真被拒绝)≥ 98%
- 误拒率(正常图被拒绝)≤ 5%

### 4.3 M3 — 物体检测与热区

**输入** 安全检查后的图
**输出** 可对话物体列表,每项含:`{id, label, bbox: {x,y,w,h}, persona_seed}`

**热区交互**
- 默认半透明描边(避免遮挡图片)
- 鼠标/手指悬停时高亮 + 显示英文物体名
- 点击触发 M4 + M5 + M6

**约束**
- 单图最多渲染 12 个热区(超出按显著性截断)
- 物体最小占图比例 ≥ 1.5%(避免点不中)

### 4.4 M4 — 2D 拟人化渲染

**渲染方式**
- 在物体边界框中合理位置(默认下方 1/3)叠加 SVG 嘴型贴图
- 默认嘴巴风格:卡通、浅色描边、可适配多种背景
- 可选 3 种嘴型预设(圆嘴/微笑/小巧),根据物体属性自动选

**Lip-sync 算法**
- TTS 音频流通过 Web Audio API 接入 AnalyserNode
- 取音频幅度(RMS)映射到嘴巴张开程度(0-100%)
- 帧率 30fps 即可,无需音素级 viseme(MVP 简化)

### 4.5 M5 — 角色人设自动生成

**输入** 物体标签 + 场景描述 + 用户水平
**输出** 角色 system prompt,含字段:
```
{
  name: "Cupcake Connie",
  personality: "甜美、爱分享烘焙故事",
  voice_style: "warm female, mid-pitch",
  catchphrases: ["Oh sweetie!"],
  topics: ["baking", "desserts", "home cooking"]
}
```

**生成规则**
- 物体类型映射音色(蛋糕→甜美女声;扳手→沙哑男声;玩偶→稚嫩童声)
- 场景上下文影响话题深度(厨房里的杯子谈烘焙;办公桌上的杯子谈职场)
- 用户水平影响词汇复杂度

**用户可调整**(最简版)
- 重新生成人设按钮
- 暂不支持自定义编辑(留 V2)

### 4.6 M6 — 语音对话引擎

**架构(全流式)**
```
用户麦克风 ──Web Speech API / Whisper──▶ 文字 ──┐
                                                  ▼
                            LLM(流式) ◀──── system prompt + 上下文
                                  │
                                  ▼
              文字气泡显示 ┐
                         ├── 同步 ──┐
              TTS 流式 ◀─┘          │
                  │                  ▼
                  ▼              用户屏幕
              扬声器 + lip-sync
```

**端到端延迟目标** < 2s(用户停止说话到 AI 开始回复)

**对话上下文**
- 系统提示词:角色人设 + 教学指令 + 用户水平
- 上下文窗口:最近 20 轮(超过自动摘要旧轮次)
- 多模态:首轮带入图片描述,LLM 知道"你是图中那个对象"

**控制按钮**
- 推流麦克风(按住说话 / 点击切换持续模式)
- 中断 AI 回复(点击屏幕任意位置或专门按钮)
- 切换到其他物体(自动结束当前对话)

### 4.7 M7 — 深度教学反馈

每轮 AI 回复包含三段(在文字气泡中分区显示):

1. **自然回应**(主气泡,语音播放)
   - 角色人设回应,自然对话
   - 隐性纠错:重述用户错误句子的正确版本

2. **学习要点**(可折叠浮窗,默认折叠)
   - 1-2 个明显错误:错误类型 + 正确写法 + 简短解释
   - 关键词汇/表达:本句中值得学的 1-2 个词或表达
   - 语法点(可选):若涉及典型语法

3. **追问练习**(语音末尾)
   - AI 主动抛出一个开放问题,推动用户继续说

**用户水平适配**
- 入门:用更简单的句式,显性纠错更频繁,关键词带中文释义
- 中阶:平衡纠错频率,关键词只给英文解释
- 高阶:几乎不显性纠错,只在严重错误时提醒,追问偏深度话题

### 4.8 M8 — 用户水平自选

- 首次进入:引导选水平(入门/中阶/高阶 + "我不确定,先聊一段试试")
- 设置入口:始终可在右上角调整
- 选"试试"模式:AI 前 3 轮估计水平,后台调整

### 4.9 M9 — 本地数据持久化

**存储方案**:IndexedDB(`personalingualive` 数据库)

**对象存储**
- `images`:`{id, thumbnail_dataurl, uploaded_at, objects[], safety_check_result}`
- `conversations`:`{id, image_id, object_id, persona, messages[], started_at, ended_at, summary}`
- `vocabulary`(预留 V2 启用):`{word, contexts[], first_seen, review_state}`
- `preferences`:`{level, ui_lang, voice_pref}`

**容量管理**
- 单图缩略图 ≤ 200KB(原图不存)
- 历史对话超 200 条时按时间清理最老的(可配置)
- 提供"导出 JSON""清除所有数据"两个明显入口

**隐私声明**(在产品中明示)
- 上传图片仅在本次会话发送给 AI 服务,不在我方服务端长期保存
- 对话内容仅存于用户本地浏览器
- 用户可随时一键清空

### 4.10 M10 — 对话总结卡

每次对话结束(点其他物体 / 关闭对话面板)生成:
- 对话时长、轮次、所聊物体名
- 学到的 3-5 个生词(LLM 摘要)
- 1-2 个语法点
- 推荐下一个练习点(过渡到 V2 任务系统)

## 5. 非功能需求

### 5.1 性能
- 首屏加载 < 2s(去除大模型调用)
- 图片安全检查 + 物体识别端到端 < 3s(P50)
- 语音对话端到端首字延迟 < 2s(P50)
- 60fps lip-sync 动画

### 5.2 兼容性
- 主目标:Chrome / Edge 最近 2 个大版本
- 次要:Safari 最近 2 版(注意 Web Speech API 差异,可能 fallback Whisper)
- 不支持:微信内置浏览器(MVP 不优化)、IE

### 5.3 可访问性(A11y)
- 文字气泡可读(对比度 ≥ 4.5:1)
- 键盘可操作核心流程(Tab/Enter/Space)
- 听障用户:支持纯文字模式(关闭语音播放,改为文字交互)

### 5.4 隐私与合规
- 图片不在我方服务端落盘,仅转发给 AI 服务后丢弃
- 对话内容不在我方服务端存储
- 首页明确告知所使用的第三方 AI 服务及其数据策略

### 5.5 安全
- 上传图片大小、格式硬限制
- API 端 rate limit(防滥用,即便无登录也按 IP 限流)
- AI prompt 注入防护(用户语音转文字后做基础清洗)

## 6. 技术架构方向(高层级)

### 6.1 前端
- 框架:React 或 Vue(开发期再定,先不锁)
- 关键库:Canvas/SVG 渲染热区与嘴型 / Web Audio API / Web Speech API / IndexedDB(idb 库)

### 6.2 后端
- 语言:Python 3.14(已定)
- 框架推荐:FastAPI(异步 + 流式 SSE/WebSocket 友好)
- 关键端点:
  - `POST /api/vision/analyze` — 接收图片,返回安全检查 + 物体列表
  - `POST /api/persona/generate` — 接收物体+场景,返回角色人设
  - `WebSocket /api/chat` — 双向流式对话
  - `POST /api/tts` — TTS 流式语音(可由 chat 端点合并)

### 6.3 AI 服务依赖
- **视觉**:支持图像理解的多模态 LLM(GPT-4o / Claude Sonnet / 国内 Qwen-VL 等),择一接入
- **对话**:同一家或独立 LLM(优先与视觉同家以省切换)
- **TTS**:ElevenLabs(音色丰富)/ Azure Speech / 国内 MiniMax,择一
- **STT**:优先 Web Speech API(免费),不可用时回退 Whisper(自托管或 API)

> 接入策略:在后端做适配层,所有 AI 服务包装为统一接口,便于切换厂商。

### 6.4 部署
- MVP:单一 Docker 容器(前端打包 + FastAPI 静态托管)
- 不需要数据库
- 不需要登录态服务
- 仅一份配置文件管理 AI Key

## 7. 关键风险与开放问题

### 7.1 风险
| 风险 | 影响 | 缓解 |
|------|------|------|
| 端到端延迟超过 2s | 对话出戏,用户流失 | 全链路流式;TTS 选择支持流式音频的厂商 |
| AI 推理成本失控 | 难以持续运营 | MVP 先打通体验,记录单次会话成本基线;V2 引入轻量本地缓存 |
| 内容安全误判过严 | 正常图被拒,体验受挫 | 双层判定:模型粗筛 + 规则白名单;提供"申诉/反馈"入口 |
| Web Speech API 浏览器差异 | 部分用户无法语音输入 | Whisper API fallback;提供文字输入兜底 |
| 嘴型贴图视觉违和 | 用户觉得"丑/怪" | 多套嘴型样式;V2 支持用户切换 |

### 7.2 开放问题
1. **AI 服务厂商选择** — MVP 期间用一家(推荐 OpenAI 或 Anthropic),还是从一开始就做适配层支持多家?(影响首版工期)
2. **UI 默认语言** — 中文?中英双语?英文 only(全沉浸)?
3. **MVP 期望开发周期** — 决定 V1 模块要不要进一步砍取(如 M10 总结卡可移到 V2)
4. **预算上限** — 决定是否能用付费高质量 TTS(ElevenLabs)还是先上免费 TTS

## 8. MVP 验收标准

### 8.1 功能验收(必须全部通过)
- [ ] 用户可上传 JPG/PNG 图片,人脸图被正确拒绝
- [ ] 上传安全图后 3 秒内显示≥ 3 个可点击热区
- [ ] 点击热区,物体长出嘴巴并主动用英文打招呼,延迟 < 2s
- [ ] 用户语音输入后,AI 在 2s 内开始流式回复
- [ ] AI 回复语音与嘴型动画同步(肉眼无明显错位)
- [ ] AI 回复中可看到至少 1 处"学习要点"折叠区
- [ ] 切换到另一个物体,新角色人设、声音明显不同
- [ ] 关闭浏览器再打开,历史对话与生词仍可查看
- [ ] 设置中"清除所有数据"能完全清空 IndexedDB

### 8.2 体验验收(主观)
- [ ] 邀请 5 位真实英语学习者(覆盖三档水平)各自体验 ≥10 分钟
- [ ] ≥ 4 位反馈"会想再用"
- [ ] ≥ 3 位反馈"AI 角色个性有趣、记得住"

### 8.3 性能验收
- [ ] P50 端到端对话延迟 < 2s
- [ ] P95 < 4s
- [ ] 首屏加载 < 2s
